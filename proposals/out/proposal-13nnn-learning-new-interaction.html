DITA 1.3 proposed feature #nn

New base and specializations for interactions that allow multiple paragraphs for question and feedback content.

Date and version information

Original requirement

Our experience with instructional designers and publishers revealed that many questions require multiple block elements for the question statement ("prompt" or "stimulus" in instructional design parlance) and for feedback. The DITA 1.2 interaction design only allows for a single paragraph question and single-paragraph feedback. This makes it impossible to use 1.2 interactions directly for such questions.

Benefits

Address the following questions:
  • Who benefits?

    Anyone who needs to represent questions and answers with multi-paragraph questions or feedback. This includes all educational publishers and many, if not most, training developers.

  • Benefit:

    The proposed markup makes it possible to use DITA learning markup directly for all questions without restriction, making it easier to adopt the use of DITA for educational publishing and instructional design by removing the need to design workarounds or non-standard vocabulary for interactions.

  • All educational publishers have this requirement, essentially without exception. Many, if not most, training groups within enterprises have this requirement. Essentially, anyone who would otherwise use the Learning and Training vocabulary very likely has this requirement.
  • This would provide a significant benefit to all users of the Learning and Training vocabulary.

Costs

Outline the impact (time and effort) of the feature on the following groups:
  • Requires update to the learningInteractionBase and learningDomain vocabulary modules (initial work completed by WEK as part of the proposal development).
  • Editors of the DITA specification:
    • Will require XX new reference topics.
    • Will require modification of 3-4 existing topics
    • No change to the general DITA architecture (although the proposal depends on the 1.3 <div> element)
  • Vendors of tools: No direct impact on vendors of tools where those tools do not provide specific processing for the current interaction markup. For tools that do provide special interaction markup processing, will need to bind the same processing to the new markup. May require adjustment to processors that depend on the current limitations to question statements and feedback.
  • This will not add to the perception that DITA is too complex. This is an incremental update to an existing vocabulary module that already has a narrow application. The workarounds required by the limitations of the 1.2 design represent greater complexity than the addition of new markup that removes the need for workarounds.

Technical requirements

Important: This section must be complete in order for the proposal to be approved.

This proposal defines two new domains for the Learning and Training vocabulary, interactionBase2Domain and learning2Domain. These two new domains mirror the existing interactionBase and learning domains, but provide a base model that allows multi-paragraph questions and multi-paragraph feedback. The new domains are otherwise semantically identical to the original domains.

Note: The names "interactionBase2" and "learning2" are the best I could think of. Suggestions for better names are welcome.
learningInteractionBase2Domain

The learningInteractionBase2Domain defines the <lcInteractionBase2> element type. This type is then the base type for a set of new specialized interaction elements in learning2Domain.

lcInteractionBase2
The <lcInteractionBase2> element differs from <lcInteractionBase> as follows:
  • It is a specialization of <div> (new in 1.3), not <fig>. This removes the content limitations that <fig> imposes (in particular, that <fig> cannot directly contain <table>) and removes the need to explicit implement special processing for lcInteractionBase2 in order to not apply default figure processing to it. The <div> element has no natural default processing semantics as it's just a wrapper element.
  • Replaces the use of <title> with a specialization of <p>, <lcInteractionLabel>. <div> does not allow <title>, so the <lcInteractionLabel> element satisfies the general requirement to have "titles" on questions without literally using the <title> element. There is some question with the Learning SC as to whether or not there is a general requirement for questions to have titles as distinct from classifying metadata that may be presented in a "titleish" way.
  • Uses <lcQuestionBase2> in place of <lcQuestionBase>. <lcQuestionBase2> allows multiple paragraphs within its content, whereas <lcQuestionBase> is itself a specialization of <p>.
  • Allows the same content as <div> whereas <lcInteractionBase> only allows figure content.
  • Makes @id optional. Not all uses of interactions require that they have IDs and it is inappropriate to require @id in all cases. Users who need to require @id can add that requirement through DITA's normal constraint facility. The implementation of this proposal includes a constraint module that makes @id required on the new interaction elements.
lcQuestionBase2
The <lcQuestionBase2> element is also a specialization of <div> and allows the same content as <div> (that is, anything that can occur in <body> except <section> or <bodydiv>).
learning2Domain
The learning2Domain is a clone of the 1.2 learningDomain. It defines new versions of each of the built-in interaction types, each of which are specialized from <lcInteractionBase2>. In addition, the content models for the interactions are extended as follows:
  • The <lcAsset> element is allowed to repeat where in DITA 1.2 there may be at most one.
  • The <lcOpenAnswer2>, <lcAnswerContent2>, <lcFeedback2>, <lcFeedbackCorrect2>, and <lcFeedbackIncorrect2> elements are all specialized from <div> and allow all <div> content.

The learning2Domain uses elements from the learningDomain when the content models of those elements are unchanged.

learningAssessment
Updated to allow both <lcInteractionBase> and <lcInteractionBase2> within <lcInteraction>.
Overall usability
This proposal makes the DITA-defined interaction vocabulary more usable by allowing a wider range of structural options and removing the need for non-standard workarounds where multi-block questions, answers, or feedback are required.
